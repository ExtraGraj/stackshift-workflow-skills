# 0.2.5 — Protocol Runtime Enforcement

## What changed

Protocol enforcement in all workflow step files now reads from `.stackshift/installed.json` rather than filtering by registry tier. Bootstrap already resolves the correct protocol selection (respecting install mode and interactive choices) and records it in `installed.json`. That resolved array is now the sole authority for runtime enforcement.

### Bugs fixed

**Bug 1 — Protocol Discovery blocks filtered by registry tier instead of `installed.json`.**
Steps 1, 2, and 4 had Protocol Discovery blocks that read `tier` directly from `_registry.json`. A user who installed `required` mode only would still have recommended protocols loaded because the filter never consulted `installed.json`. Fixed: all enforcement blocks now check the `protocols` array in `installed.json` and skip any protocol whose `id` is absent from that array.

**Bug 2 — Dead `appliesTo` filter in all step files.**
The discovery blocks said `where appliesTo includes Step N`, but no `appliesTo` field exists in any registry entry. This filter was silently a no-op — the bulleted protocol lists below it were doing the actual work. Fixed: the broken filter language is removed; each step file now enumerates its relevant protocols explicitly.

**Bug 3 — Step 4 optional protocol check read the registry, not `installed.json`.**
Step 4 listed optional protocols as "active in the merged registry" — a registry check, not an install check. A user who did not select `accessibility`, `brand`, or `claude-design-handoff` at install time would still see them applied. Fixed: Step 4's discovery block now uses the same `installed.json`-based pattern as Steps 1 and 2.

### New: Cross-Cutting Optional Protocols in SKILL.md

`modal-sheet` and `auto-verify-hook` cannot be tied to a single step and were previously invisible to the workflow unless verbally prompted. SKILL.md now includes a two-stage cross-cutting discovery section:

1. **Stage 1 — `installed.json` check:** If the protocol's `id` is in the `protocols` array, load and apply proactively on every invocation.
2. **Stage 2 — Keyword discovery:** If the `id` is not in the array but the user's request matches a keyword, load the protocol and surface a visible notice.

### Registry and schema change

Added a `keywords` array to `auto-verify-hook` and `modal-sheet` entries in `protocols/_registry.json`. The shared `_registry.schema.json` was updated to declare `keywords` as an optional `array of string` property on registry entries — the existing `additionalProperties: false` constraint would otherwise reject the new field and surface a schema validation warning in IDEs.

### Graceful degradation

If `.stackshift/installed.json` is missing, has no `protocols` array, or the array is not valid JSON, enforcement is a no-op and the workflow continues — a single warning is surfaced. The skill never throws or blocks on a parse failure.

## What does not change

- Protocol lookup location: project copy in `.stackshift/protocols/` still takes precedence over skill fallback.
- Bootstrap behavior: tiers still drive install mode selection. Enforcement blocks read what bootstrap already recorded.
- Seed architecture: gate pattern in Step 2 is already correct and was not modified.
- Tier bundle SKILL.md files: remain as reference lookup tables, unchanged.
