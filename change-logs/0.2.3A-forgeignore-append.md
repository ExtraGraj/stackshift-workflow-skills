# 0.2.3A — Bootstrap `.forgeignore` Append Fix

**Released:** 2026-05-06

---

## Summary

This sub-release fixes the Bootstrap Step 8 `.forgeignore` handling to append StackShift entries when the file already exists, preventing data loss and avoiding the need for manual file reconciliation in projects with existing ignore rules.

---

## B-3 · `.forgeignore` append-on-materialize behavior

**Affects:** `cli/src/writer.ts`, `skills/stackshift-core/bootstrap/install.md`

### Previous behavior

When `.forgeignore` already existed in a project, the bootstrap step would skip writing it entirely. Projects that had custom ignore rules would lose StackShift's defaults (Sanity/Next.js build artifacts, UI Forge regeneratable design files).

### New behavior

- **If `.forgeignore` does not exist:** Create it with full StackShift defaults (Sanity + Next.js + UI Forge entries).
- **If `.forgeignore` already exists:** Append StackShift defaults to the file. Sections are intelligently detected to avoid duplicates:
  - If the file already has "StackShift — Sanity + Next.js defaults" comment, those entries are skipped.
  - If the file already has "UI Forge — Claude Design integration" comment, those entries are skipped.
  - Only missing sections are appended.
- **User-authored content is always preserved** — no lines are removed or reordered.

This applies to both materialization modes:
- `npx stackshift init --bootstrap` (interactive)
- `npx stackshift init --bootstrap --no-interactive` (non-interactive)

### Rationale

Projects often ship with their own `.forgeignore` (or `.gitignore`) files to handle environment-specific or tooling-specific patterns. Overwriting or skipping the file led to:
1. Lost user configurations on fresh bootstrap
2. Stale ignore entries that no longer matched project structure after upgrades
3. Requiring manual file edits to reconcile StackShift defaults with project rules

The append approach respects existing configuration while ensuring StackShift essentials are always present.
