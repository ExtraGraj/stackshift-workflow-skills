# 0.2.5A — Audit Fixes

## What changed

Eleven low-complexity fixes from the 2026-05-07 project audit, covering documentation discrepancies, dead code removal, and simple code corrections. No architectural changes, feature additions, or refactors.

### Bugs fixed

**Fix 1 — Stale path docs (`.stackshift/protocol/` → `.stackshift/protocols/`).**
The directory was renamed from singular to plural in v0.2.3. Three files still referenced the old singular path:
- `CLAUDE.md` (lines 47, 73): `/docs/protocol/` → `.stackshift/protocols/`
- `README.md` (line 430): file-structure tree showed `protocol/` (singular) — corrected to `protocols/`
- `skills/stackshift-core/protocols/_registry.json` (line 3): description field path corrected

**Fix 2 — Recommended-bundle protocol count off by one.**
`README.md` stated `4 required + 6 recommended` (line 69) and `4 required + 5 recommended` (line 578). The registry has 7 recommended protocols. Both counts corrected to `7`.

**Fix 3 — Dead `--materialize` flag removed.**
`cli/src/flags.ts` parsed `--materialize` and set `flags.materialize = true`, but nothing in the codebase read it. The default behavior already runs materialization unless `--no-materialize` is passed. Removed: parser case, Flags-interface field, help-text row, and README example.

**Fix 4 — `modal-sheet` missing from optional-protocol list.**
`README.md` line 458 listed optional protocols as `brand`, `claude-design-handoff`, `auto-verify-hook` but omitted `modal-sheet`. Added to the parenthetical.

**Fix 5 — `hasRequiredFlags()` renamed to `isNonInteractive()`.**
The function did not check whether required flags were present — it simply returned whether `--no-interactive` was passed. Renamed to `isNonInteractive` for clarity. Import updated in `install.ts`.

**Fix 6 — Unreachable custom-tier validation removed.**
`flags.ts` had a dead check for custom tier values that could never appear (already constrained by `parseFlags`). Removed the unreachable error branch.

**Fix 7 — `--bootstrap` flag removed.**
The deprecated `--bootstrap` flag was parsed but pointed users at the equally dead `--materialize` flag. Removed: parser case, Flags-interface field, help-text row, and the conditional outro message in `install.ts`.

**Fix 8 — `npx stackshift init` standardized to `npx @extragraj/stackshift-skills init`.**
`CLAUDE.md` (lines 9, 25) and `README.md` (lines 332, 361) used `npx stackshift init` which only works after a local/global install. Changed to the published package form `npx @extragraj/stackshift-skills init` for consistency with the rest of the docs.

**Fix 9 — Inconsistent install command advice in `repair.ts`.**
`repair.ts` line 406 advised `npx skills add extragraj/stackshift-workflow-skills to reinstall` while line 348 used `npx @extragraj/stackshift-skills init`. Standardized on the CLI command for in-command guidance.

**Fix 10 — `materializationDone` branch missing from SKILL.md bootstrap check.**
`skills/stackshift-core/SKILL.md` Section 0.2 only branched on `bootstrapRequired`. Added a third bullet: `Present with "materializationDone": true → run bootstrap/install.md to complete UI Forge integration steps (Step 6+) only`.

**Fix 11 — SKILL.md trimmed: removed Section 6, removed Hard Rules, trimmed Section 4.**
Three documentation reductions in `skills/stackshift-core/SKILL.md`:
- **Section 6 "Extending the Skill" removed** — skill-maintainer documentation (how to add protocols, workflow steps, references to the skill itself). Not relevant to the agentic model.
- **Section 7 "Hard Rules" removed** — all 8 rules were redundant, already enforced through workflow steps, protocols, or references (step ordering in Section 1; Sanity v4+ APIs in `references/versions.md`; no `any` in `references/versions.md`; no scalar GROQ projections in `workflow/5-groq.md`; no fallback variant in `protocols/variant-router.md`; no `@stackshift-ui` imports in `protocols/variant-router.md`; no duplicate field names in `protocols/sub-field-visibility.md`; no `defineField()` wrappers in `protocols/factory-function-pattern.md`).
- **Section 4 "Adding Custom Protocols" trimmed** — replaced step-by-step creation instructions with a concise agent-centric explanation of how custom protocols work and when to apply them. Added a brief note pointing to `_registry.schema.json` if the user asks to create one.
