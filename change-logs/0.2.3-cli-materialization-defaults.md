# 0.2.3 — CLI Materialization Defaults & Bootstrap Clarification

## Changes

### 1. Materialization Now Default Behavior
- **CLI default:** `npx stackshift init` now materializes protocols immediately by default
  - Materializes to `.stackshift/protocols/`
  - Creates project registries and design standards
  - Sets `"materializationDone": true` in marker
- **New flags:**
  - `--materialize` — Explicit (redundant with default, for clarity)
  - `--no-materialize` — Defer all steps to AI agent (sets `bootstrapRequired: true`)
- **Deprecation:** `--bootstrap` flag still works but is deprecated; use `--materialize` instead

### 2. Directory Rename: `.stackshift/protocol/` → `.stackshift/protocols/`
- Clarifies that directory contains *multiple* protocols
- Updated all path references in CLI and documentation
- Bootstrap/install.md updated to reflect new structure

### 3. File Rename: `stackshift-ui.md` → `stackshift-section-variants.md`
- Clarifies purpose: conventions for generating section *variants*, not general UI
- Constant renamed from `STACKSHIFT_UI_MD` to `STACKSHIFT_SECTION_VARIANTS_MD`
- Updated all path references

### 4. Enhanced Project Protocol Registry
- Registry now populated with `materialized` array listing selected protocols
- Added discovery guide: `.stackshift/protocols/README.md` created during materialization
- Custom protocols can be added to `custom` array in registry

### 5. Bootstrap Documentation Clarifications
- Step 1: Updated to check for `materializationDone` field
- Step 6: Added note that Steps 6f (CLI) and 6a–6e (AI) may overlap
- Step 6a: Added explanation of variant-router vs section-variants conventions
- Step 6f: Clarified as hybrid (runs in both CLI and AI phases)
- Step 8: Added idempotency note and UI Forge entry explanation

## Breaking Changes

- **CLI default behavior changed:** Materialization now runs by default instead of deferring to AI
- **Directory name changed:** Projects must rename `.stackshift/protocol/` to `.stackshift/protocols/`
  - Old projects with `bootstrapRequired: true` marker continue to work
  - Agent detects and handles legacy structure
- **Marker field changed:** `bootstrapRequired` now coexists with `materializationDone`
  - Default (materialization runs): Write `materializationDone: true`
  - With `--no-materialize`: Write `bootstrapRequired: true`

## Migration Notes

- Existing projects with `.stackshift/protocol/` directory will need manual rename or re-run `init` with `--materialize`
- Legacy `bootstrapRequired: true` marker still supported; agent detects and processes correctly
- No changes to protocol content or workflow steps — only infrastructure and bootstrap timing
