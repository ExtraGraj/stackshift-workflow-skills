# 0.1.8 — UI Forge Integration Expansion

**Release date:** 2026-04-24
**Previous version:** 0.1.7
**Target:** UI Forge ≥ 0.1.8

## Summary

Brings StackShift's skill system up to date with UI Forge features shipped between v0.1.3 and v0.1.8. Adds two new protocols that activate UI Forge signals, replaces hand-rolled contract checks with UI Forge's dedicated validators, emits `@contract-version` on every props interface, and improves bootstrap ergonomics with design standards seeding and scan exclusions.

All changes are additive. Projects on StackShift 0.1.7 continue to work unchanged against any UI Forge ≥ 0.1.2.

---

## New Protocols

### Accessibility (recommended tier)

**Files:** `skills/stackshift-core/protocols/accessibility.md`, `skills/stackshift-core/protocols/_registry.json`

A new recommended-tier protocol that activates UI Forge's `SIGNAL_A11Y`. When selected during bootstrap, the AI agent writes `a11yRequired: true` to `.stackshift/installed.json`. UI Forge's paired-mode detection reads this marker and automatically composes WCAG 2.1 AA rules — semantic HTML, accessible names, focus visibility, reduced-motion handling — into every variant generation for the project.

No per-invocation flag is needed once the protocol is installed. StackShift owns activation; UI Forge owns enforcement.

**Step 4 postcondition (when active):** Generated FORGE NOTES header must contain an `A11Y` sub-block.

### Brand (optional tier)

**Files:** `skills/stackshift-core/protocols/brand.md`, `skills/stackshift-core/protocols/_registry.json`

A new optional-tier protocol that registers a project brand document so UI Forge's role classifier composes `SIGNAL_BRAND` during variant generation. Bootstrap writes `designStandards.brand: "./design/standards/brand.md"` to `design/design-arch.json` and creates a starter brand document at that path (skipped if the file already exists).

Not installed by default — only relevant when the project has brand guidelines to enforce.

## Contract Handshake Improvements

### `@contract-version` JSDoc tag on props interfaces

**Files:** `skills/stackshift-core/protocols/variant-router.md`, `skills/stackshift-core/workflow/3-types.md`, `skills/stackshift-core/workflow/4-variants.md`, `skills/stackshift-core/references/versions.md`

Every section props interface now carries a `/** @contract-version 1.0.0 */` JSDoc tag immediately above the `export interface` declaration. UI Forge 0.1.3+ parses this tag and includes it in the CONTRACT header of generated output, creating an auditable record of which contract version a variant was generated against.

Current contract version: `1.0.0`. The version tracking entry in `references/versions.md` documents when and how to bump it in coordination with UI Forge's `SUPPORTED_CONTRACT_VERSIONS`.

### Pre-flight input validation (`--validate-input`)

**File:** `skills/stackshift-core/workflow/4-variants.md`

Step 4d now runs a pre-flight validation step (4d.1) before the main UI Forge invocation:

```bash
node ${UI_FORGE_SKILL_DIR}/scripts/invoke.js \
  --validate-input \
  --signal CONVERT_VARIANT \
  --refs <path-to-types.ts>
```

On success, UI Forge prints `ui-forge: input validation passed — interface: <Name> (<path>)` and generation proceeds. On failure, Step 4 halts with the stderr output. This catches malformed props interfaces before spending a full generation on a broken handoff.

### Postcondition validation via `validate-contract.js`

**File:** `skills/stackshift-core/workflow/4-variants.md`

The manual postcondition checklist is replaced with a tiered structure:

- **Structural (validator-driven):** `node ${UI_FORGE_SKILL_DIR}/scripts/validate-contract.js <output> <contract>` must exit 0. Covers exports, contract import, null handling, and props interface integrity.
- **StackShift-specific (manual):** FORGE NOTES header present; `index.tsx` bytewise unchanged; no unexpected files.
- **Accessibility (when protocol active):** FORGE NOTES header contains `A11Y` sub-block.

### Failure Modes table — `Checker` column

**File:** `skills/stackshift-core/workflow/4-variants.md`

The Failure Modes table gains a `Checker` column distinguishing UI Forge-owned detection (pre-flight validator, post-gen validator, stderr) from StackShift-boundary checks (FORGE NOTES header, `index.tsx` diff, unexpected files). This makes it clear which tool to investigate when a failure occurs.

## Bootstrap Improvements

### Design standards directory seeded at bootstrap

**File:** `skills/stackshift-core/bootstrap/install.md`

Bootstrap Step 6 now includes a new sub-step (6f) that seeds `design/standards/`:

- Creates `design/standards/` if absent.
- Writes `design/standards/stackshift-ui.md` (skipped if file exists) containing the StackShift UI conventions UI Forge reads via its three-layer design-standards resolver. This replaces the protocol-path bridge as the primary `stackshiftComponentStandard` reference, while keeping the protocol-path as a fallback for projects that pre-date v0.1.8.
- If the `brand` protocol is selected, writes `design/standards/brand.md` (skipped if file exists) as a starter brand document.

### `.forgeignore` emitted at bootstrap

**File:** `skills/stackshift-core/bootstrap/install.md`

A new Step 8 writes a `.forgeignore` file at the project root if one does not exist. The defaults exclude Sanity Studio, Sanity internals, Next.js build output, and common build directories — directories StackShift projects always have and `scan.js` should never walk. The file is never overwritten after first write.

### `a11yRequired` field in `installed.json`

**File:** `skills/stackshift-core/bootstrap/install.md`

Bootstrap Step 7 now conditionally writes `a11yRequired: true` to `.stackshift/installed.json` when the `accessibility` protocol is in the materialized set. Absence of the field means "not requested" — consistent with UI Forge's expectation.

### Paired-mode constraints documented in Step 4

**File:** `skills/stackshift-core/workflow/4-variants.md`

A brief "Paired-mode constraints" notice is added near the top of Step 4, stating that `--creative`, `--diff`, and `--preview` are refused by UI Forge when invoked in paired mode. This prevents users from hitting confusing errors after reading UI Forge's README.
